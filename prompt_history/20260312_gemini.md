# 日次AIレビューシステム — 実験概要・ポイント・実装案

## 1. 実験の目的

身内の個人メディアに対して、3つのAIモデル（Claude / Gemini / GPT）が**毎日自動で順次レビュー・ディスカッション**を行い、その議論の蓄積から**時間経過とともに創発的な知性が現れるか**を検証する。

## 2. サイトの特殊性（分析軸の前提）

このサイトは**通常のメディアとは存在目的が根本的に異なる**。従来のGA4分析フレームワーク（直帰率改善、ファネル最適化）をそのまま適用すると、的外れな結論が出るリスクがある。

| 普通のサイト | 対象の個人メディア |
|---|---|
| コンテンツは手段（CVのため） | コンテンツ自体が目的 |
| 隠す（ノウハウ = 競争優位） | **全部見せる**（プロンプトまで公開） |
| PV・CVR・LTVが重要 | **「誰が何を再現したか」** が重要 |
| SEO最適化 | **LLMO**（AI学習ソースになる戦略） |
| 離脱率が悪 | 1記事で完結して離脱するのが正常 |
| 有料課金が主 | 22記事中21記事が無料 |

### AIレビュアーに持たせるべき分析軸

1. **再現可能性**: 読者が記事を読んで「やってみる」ことが可能か
2. **シリーズの発展**: 深まり（3部作化）と横展開（新テーマ）のバランス
3. **技術的正確性**: 特にセキュリティ関連記事の正確さ
4. **AI共創の自己言及性**: AIが作ったサイトの記事をAIがレビューする構造
5. **LLMO効果**: AIが参照しやすい構造・記述になっているか

## 3. アーキテクチャ

### 全体構成

```
EventBridge (cron: 毎朝6:00)
    │
    ▼
Step Functions（合計 15-25分）
    │
    ├── ① Lambda → AgentCore Runtime [Gemini Agent]     ← 約3-5分
    │      input:  サイトHTML + GA4データ + 直近3日レビュー(L1)
    │      output: Geminiレビュー
    │      → DynamoDB保存
    │
    ├── ② Lambda → AgentCore Runtime [GPT Agent]        ← 約3-5分
    │      input:  同上 ＋ ①Geminiレビュー
    │      output: GPTレビュー（Geminiに対する同意・反論含む）
    │      → DynamoDB保存
    │
    ├── ③ Lambda → AgentCore Runtime [Claude Agent]     ← 約3-5分
    │      input:  同上 ＋ ①Geminiレビュー ＋ ②GPTレビュー
    │      output: Claudeレビュー（2者に対する同意・反論含む）
    │      → DynamoDB保存
    │
    ├── ④ Lambda → AgentCore Runtime [Swarm Agent]      ← 約5-10分
    │      input:  ①②③全結果 → SharedContext
    │      output: 3者ディスカッション（深掘り・合意形成）
    │      → DynamoDB保存
    │
    └── ⑤ Lambda → S3 put_object（HTML公開）             ← 数秒
              → DynamoDB（公開メタデータ）
```

### 設計上の重要判断

#### Lambda は「封筒を渡す係」に徹する

- Lambda 自身はエージェントを実行しない
- AgentCore Runtime の API を呼び出して結果を受け取るだけ（数秒〜数分の待ち）
- 15分制限は各ステップで余裕あり。合計実行時間はStep Functionsが管理

#### 順次実行の意味

並列（独立レビュー）だと「3つの似たような分析レポート」になりがち。順次（前の人の意見を踏まえて発言する）にすることで、回を重ねるごとに視点が研ぎ澄まされる。

```
① Gemini: 「この記事のLLMO効果は高い。プロンプト全公開が差別化要因」

② GPT:   「Geminiの指摘に同意。ただ再現可能性の観点で、
            環境構築手順にWSL前提が多く、Mac/Windowsユーザーの
            離脱リスクがある」

③ Claude: 「GPTの環境問題の指摘は妥当。ただGA4のスクロール率を見ると
            読了率は高い。問題は「到達率」ではなく「流入元の偏り」では。
            検索クエリ分析が必要」

④ Swarm:  （3者の対立点を深掘り → 合意事項整理 → アクションアイテム導出）
```

#### ④ Swarm の役割

①②③で3者の意見は出揃っている。④の役割は：

- 3者の**意見の対立点**を特定し、掘り下げる
- **合意事項**をまとめる
- **具体的アクションアイテム**を導出する
- 翌日のL1に入れる**要約**を生成する

構造的には、①②③が個別レビュー、④がエディターの編集会議。

## 4. 記憶戦略（L1のみスタート）

### 設計方針

- **L1のみ（直近3日の全文）** から開始する
- L2（週次ダイジェスト）/ L3（RAG検索）は後から追加判断

### L1で始める理由

1. **ベースライン確立**: L1のみの出力品質がわかって初めて、L2/L3の価値が測れる
2. **コスト感覚**: 3モデル × 3日分全文のトークン消費量を実数で把握
3. **YAGNI**: L1で十分かもしれない。不十分だと判明してからL2/L3を足す

### トークン量の見積もり

- 1日のレビュー出力: 約5,000-10,000トークン（4ステップ合計）
- L1（3日分）: 最大30,000トークン
- 各モデルのコンテキスト上限に対して余裕あり

## 5. エージェント構成

### 各エージェントが持つツール

| ツール | 種類 | 用途 |
|---|---|---|
| `fetch_site_content` | @tool | サイトHTMLを取得・テキスト抽出 |
| `get_ga4_metrics` | @tool | GA4 APIから記事別メトリクス取得 |
| `get_past_reviews` | @tool | DynamoDBから直近3日のレビュー全文取得 |
| `save_review` | @tool | レビュー結果をDynamoDBに保存 |
| GitHub | MCP (PAT認証) | リポジトリ内のコード・プロンプト・コミット履歴を閲覧 |

### ツール選択の判断

- **GA4**: MCP不要。バッチ用途なので `@tool` で GA4 API を直叩きすれば十分
- **GitHub**: MCP推奨。ファイルツリー走査・コード検索・コミット差分など、fetchでは非現実的な操作が多い。PAT認証で十分（3LO OAuth不要）
- **Python（Code Interpreter）**: 不要。AIの役割はレビュアーであり開発者ではない

### 目的案1：コンテンツとしてのエンタメ化（「AI編集会議」の公開）
**目的：AI同士のガチ議論自体を、サイトの「裏名物コンテンツ」にする**

*   **概要:** 記事を読みに来た読者が、「この記事に対してAIたちはどう評価しているのか？」を楽しむための設計です。
*   **面白くする工夫:**
    *   各モデルに強烈な「ペルソナ（性格）」を持たせます。
        *   **Gemini:** 読者フレンドリーでミーハー、Googleのエコシステムにこだわる。
        *   **GPT:** ゴリゴリのエンジニア気質で、論理の飛躍やツールの非効率さをチクチク指摘する。

    *   ④のSwarm（ディスカッション）の結果を、**対話形式（チャットUI）で記事の末尾や専用ページに毎日自動公開**します。「GPTが著者のコードにダメ出しし、Claudeが擁護する」といったプロレスが起きるのをエンタメとして楽しみます。

### 目的案2：究極の「LLMO（AI向け最適化）」のフィードバックループ構築
**目的：サイトを「世界で最もAIが読みやすい・学習しやすいメディア」に進化させる**

*   **概要:** 対象のサイトは「LLMO（AI学習ソースになる戦略）」を重視しています。であれば、AI自身に**「我々AIにとって、今日の記事は学習しやすかったか？」**を毎日採点させ、改善案を出させます。
*   **面白くする工夫:**
    *   「ここは画像で説明されているので、我々（テキストベースのAI）は文脈を補完しきれなかった。altタグかマークダウンの表にしてほしい」など、**AI視点でのダメ出し**をもらいます。
    *   人間（あなた）は、そのAIの指摘に従ってサイトの構造や記事の書き方をチューニングしていきます。「AIに育てられるメディア」という実験そのものが、唯一無二の価値（面白さ）になります。

### 目的案3：「自律進化型メディア」への移行（次期ロードマップの自動生成）
**目的：著者のアイデア出しを不要にし、AIに「次に書くべき記事」を指定させる**

*   **面白くする工夫:**
    *   AI会議の結論として、「昨日の記事はMacユーザーへの配慮が欠けていたから、明日は『Windows/Mac両対応の開発環境構築』を書くべき」「ClaudeとGPTでセキュリティに関する見解が割れたので、著者はこれを検証する記事を書け」といった**指示書（アクションアイテム）**を出させます。
    *   「AIの指示に従って人間が記事を書く」という、通常とは逆の主従関係を楽しむことができます。

### 目的案4：「生きたポートフォリオ」としての技術デモ

 * CloudFront Functions 用 テンプレート
 * 
 * 処理を「ルール」として定義し、Arrayに追加することで適用順序を制御します。
 */
function handler(event) {
    var request = event.request;
    var clientIp = event.viewer.ip; // 接続元のIPアドレス

    // --- 設定ここから ---

    // 1. 【パス除外ルール】特定のパス（Webhookなど）は以降の全チェックをスキップしてオリジンへ通す
    var skipAuthRule = function(req, next) {
        var excludePaths = ['/webhook_path/', '/api/public/'];
        for (var i = 0; i < excludePaths.length; i++) {
            if (req.uri.startsWith(excludePaths[i])) {
                return req; // 以降の処理をせず、オリジンへ直接送信
            }
        }
        return next(req);
    };

    // 2. 【緊急ブロックルール】特定のIPやユーザーエージェントを拒否（WEBアタック対策）
    var emergencyBlockRule = function(req, next) {
        var blockedIps = ['1.2.3.4']; // 攻撃元のIPをここに追記
        var blockedUa = 'BadBot';     // 攻撃元のUAの一部をここに指定

        var ua = (req.headers['user-agent'] && req.headers['user-agent'].value) || '';

        if (blockedIps.indexOf(clientIp) !== -1 || ua.indexOf(blockedUa) !== -1) {
            return {
                statusCode: 403,
                statusDescription: 'Forbidden'
            };
        }
        return next(req);
    };

    // 3. 【エンドポイント無効化】脆弱性が発見されたパスを即座に遮断（応急処置）
    var killSwitchRule = function(req, next) {
        var vulnerablePaths = ['/login/credential_stuffing_target', '/search?q=xss'];
        for (var i = 0; i < vulnerablePaths.length; i++) {
            if (req.uri === vulnerablePaths[i]) {
                return {
                    statusCode: 404, // 存在しないことにする
                    statusDescription: 'Not Found'
                };
            }
        }
        return next(req);
    };

    // 4. 【メンテナンスルール】503を表示。ただし関係者IPは除外
    var maintenanceRule = function(req, next) {
        var isMaintenance = false; // 有効にする場合は true に
        var allowIps = ['111.222.333.444']; // 関係者のIP

        if (isMaintenance && allowIps.indexOf(clientIp) === -1) {
            return {
                statusCode: 503,
                statusDescription: 'Service Unavailable',
                headers: {
                    'content-type': { value: 'text/html; charset=UTF-8' },
                    'retry-after': { value: '3600' }
                },
                body: {
                    encoding: "text",
                    data: `<!DOCTYPE html>
<html lang="ja">
<head>
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width">
<title>システムメンテナンスのお知らせ</title>
</head>
<body>
<h1>システムメンテナンスのお知らせ</h1>
</body>
</html>`,
                }
            };
        }
        return next(req);
    };

    // 5. 【リダイレクトルール】QR間違い、Googleリンク切れ(301)の高速修正
    var redirectRule = function(req, next) {
        var redirectMap = {
            '/old-qr-path': 'https://example.com/new-path',
            '/category/114': 'https://example.com/category/115',
            '/item/0000001': 'https://example.com/item/0000006'
        };

        if (redirectMap[req.uri]) {
            return {
                statusCode: 301,
                statusDescription: 'Moved Permanently',
                headers: {
                    'location': { value: redirectMap[req.uri] }
                }
            };
        }
        return next(req);
    };

    // 6. 【Basic認証ルール】開発環境などの保護
    var basicAuthRule = function(req, next) {
        var authHeader = req.headers['authorization'];
        var expectedValue = 'Basic dXNlcjpwYXNz'; // user:pass をbase64化したもの

        if (!authHeader || authHeader.value !== expectedValue) {
            return {
                statusCode: 401,
                statusDescription: 'Unauthorized',
                headers: {
                    'www-authenticate': { value: 'Basic' }
                }
            };
        }
        return next(req);
    };

    // --- 実行処理（適用したい順に並べる） ---

    var processes = [
        skipAuthRule,       // 最初にWebhookなどを逃がす
        emergencyBlockRule, // 次に攻撃者を弾く
        killSwitchRule,     // 脆弱なパスを塞ぐ
        maintenanceRule,    // メンテナンスなら止める
        redirectRule,       // リダイレクト対象なら飛ばす
        basicAuthRule       // 最後にBasic認証
    ];

    // 各関数をチェーン状に結合するラッパー
    var wrapper = processes.reverse().reduce(function(next, currentProcess) {
        return function(req) {
            return currentProcess(req, next);
        };
    }, function(req) {
        return req; // すべてのチェックを通過した場合はオリジンへ
    });

    return wrapper(request);
