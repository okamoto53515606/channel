## 1. Lambda順次実行 → Fargate + Swarm自律議論：**賛成。大幅に設計がシンプルになる**

現行blueprintの最大の複雑性は「ローカルはSwarm、本番は4分割Lambda + Step Functions」という二段構え（§5, §6, §10）にあります。Fargate化でこの乖離がなくなるのは大きい。

**メリット:**
- ローカルと本番で**同一のSwarmコード**が走る。デバッグ→本番の差分がほぼゼロ
- Step Functionsのstate passing設計（§10の冗長なJSON定義）が丸ごと不要に
- `handoff_to_agent`による自律議論は、順次実行の「前の人の意見を踏まえてレス」より**はるかに面白い掛け合い**が期待できる。"あーでもないこーでもない"のプロレスはSwarmの本領
- 1時間のタイムアウトなら、コンテキストをケチる必要がなく、過去スレ+GitHubコード+画像をフルに食わせられる

**注意点:**
- Fargateタスクは起動に30-60秒のコールドスタートがある。毎朝1回のバッチなので問題にはならないが、コスト概算（§15）は「AgentCore Runtime ~$5 + Lambda ~$1 + Step Functions ~$0.01」→ 「Fargate タスク起動時間課金」に書き換え。Fargate Spot使えば安い
- **異常時のリトライ粒度が粗くなる**。Step Functionsなら「GPTだけ失敗→GPTだけリトライ」ができたが、Fargateなら全体やり直し。ただし毎朝1回のバッチなので許容範囲
- EventBridge → Fargate起動は `aws ecs run-task` をEventBridgeターゲットにするか、EventBridge → Lambda(薄い) → `run-task` のパターンが一般的

| **`http_request`** | `convert_to_markdown=True` オプションで記事HTMLをクリーンなMarkdownに変換取得できる。現在の `fetch_article_content` @tool を簡素化 or 置き換え可能 |
| **`current_time`** | バッチ実行時のタイムゾーン制御。`timezone="Asia/Tokyo"` でJST取得。地味だがBBSの日付ヘッダー生成に便利 |
| **`rss`** | okamoのhomepageにRSSフィードがあれば、`fetch_article_list` @tool の代替・補完になる。新着記事の検出がシンプルに |
| **`batch`** | 複数ツールの並列実行。例: 記事取得 + GitHub閲覧 + 画像読取りを同時に走らせ、Swarmのターンあたりの時間を短縮 |

### B. 検討の価値あり

| ツール | 理由 |
|---|---|
| **`tavily_search`** | AIエージェント向けに最適化されたWeb検索。記事内の技術的主張のファクトチェックや、関連情報の補足に使える。Claude（エンジニア）が「この設計パターン、本当にベストプラクティスか？」と裏取りする武器になる（要 TAVILY_API_KEY） |

| **`think`** | 複雑な推論を分岐させて深く考えるツール。スコアリングの根拠付けやまとめ時にClaudeが使えば、議論の質が上がる可能性あり |
| **`journal`** | 構造化されたタスクログ。Swarm議論のプロセスを記録し、デバッグやレビュー品質のトラッキングに活用 |

### C. 不要と判断

| ツール | 理由 |
|---|---|
| `shell` / `python_repl` / `code_interpreter` | レビュアーであり開発者ではない（blueprintの方針通り） |
| `browser` | Fargate上でChromium起動はオーバーヘッド大。`http_request` + `tavily_extract` で十分 |
| `slack` / `speak` / `use_computer` | ユースケース外 |

| `generate_image` / `nova_reels` | BBSテキスト生成がメイン。画像生成は不要 |

echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \

sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

newgrp docker

## Bedrock で Claude を使うために必要なこと（2026年3月時点）

### 1. モデルアクセス許可 — **自動化された（大幅簡略化）**

2025年10月以降、**Bedrock のサーバーレスモデルは全て自動で有効**になりました。旧来の「Model Accessページで手動で有効化ボタンを押す」手順は **廃止** されています。

ただし **Anthropic モデルだけ例外** で、**1回だけ use case フォームの送信が必要**です。

### 2. Anthropic 初回利用フォーム（1回きり）

| 項目 | 内容 |
|---|---|
| **何をする** | Anthropic の利用目的フォームを1回送信する |
| **頻度** | AWSアカウントごとに1回。Organizations を使っている場合は管理アカウントで1回送信すればメンバーアカウント全部に継承 |
| **方法（コンソール）** | Bedrock コンソール → モデルカタログ → 任意の Anthropic モデルを選択 → フォーム入力・送信 |
| **方法（API/CLI）** | `PutUseCaseForModelAccess` API を呼ぶ（`formData` にBase64エンコードしたフォームデータを渡す） |
| **送信後** | **即座に**アクセス可能になる（待ち時間なし） |

```bash
# CLI でフォーム送信する場合（boto3 でも同等）
aws bedrock put-use-case-for-model-access \
  --form-data "$(echo -n '{"company_name":"okamo","use_case":"AI-powered blog review system"}' | base64)" \
  --region us-east-1
```

### 4. Cross-Region Inference Profile について

blueprintで `BEDROCK_MODEL_ID=us.anthropic.claude-opus-4-6-v1` と `us.` プレフィックスを使っているため、これは **US Geographic Cross-Region Inference Profile** です。

| 項目 | 内容 |
|---|---|
| **プレフィックス `us.`** | US リージョン内（us-east-1, us-east-2, us-west-2）でルーティング |
| **メリット** | 単一リージョンより高スループット。追加コストなし |
| **IAM の注意** | inference profile ARN **と** 各 destination リージョンの foundation-model ARN の両方に `Allow` が必要 |

Cross-Region を使う場合のIAMは上記ポリシーでカバー済み（`arn:aws:bedrock:*:210387976006:inference-profile/us.anthropic.*` と `arn:aws:bedrock:us-east-1::foundation-model/anthropic.claude-*`）。
